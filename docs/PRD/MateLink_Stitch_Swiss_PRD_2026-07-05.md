> 项目来源：Stitch 项目 `MateLink - Tesla 监控 (简约白)` (`projects/11493757920836657212`)
> 版本：v1.0 Draft
> 日期：2026-07-05
> 作者：Codex for Jovi
> 适用范围：`app_mimo` 移动端主产品需求梳理，重点覆盖移动端页面、信息架构、分析能力、设置链路与实现边界

## Problem Statement

当前仓库已经同时存在 Android、iOS、Web 三套 MateLink 实现和多份历史 PRD，但“移动端最终要交付什么体验”仍然分散在 Stitch 页面、旧 PRD、README 和各端代码中。对于后续开发而言，缺少一份直接从页面集合反推出来的、可用于排优先级和落实现模块的产品需求文档。

从用户视角看，问题不是“功能点不够多”，而是：

- 已经部署 TeslaMate 的 Tesla 车主，希望在手机上以原生、稳定、低干扰的方式查看车辆状态，而不是反复打开 Grafana 或桌面端后台。
- 用户需要的不只是当前电量和位置，还需要行程、充电、能耗、续航、电池健康、待机耗电、热力分布等一整套围绕 TeslaMate 数据的移动分析体验。
- 页面之间需要形成自然的钻取链路：总览看趋势，列表看记录，详情看曲线，分析页看建议，设置页管连接、主题、语言和模拟数据。
- 仓库里已有实现并不完全一致，尤其导航结构、分析深度、图表来源、地图能力、通知和多实例策略仍有分歧，需要一个“以页面为依据”的统一产品口径。

## Solution

基于 Stitch 的白色简约版移动项目，将 MateLink 明确为一个“面向已部署 TeslaMate 用户的移动数据监控与分析应用”，并以四层能力组织产品：

1. 实时总览：在仪表盘中给出车辆当前状态、关键硬件状态、短期能耗与最近活动。
2. 历史与详情：围绕行程和充电建立列表、摘要、单次详情与曲线分析链路。
3. 深度分析：提供热力图、能耗分析、续航分析、电池健康、待机耗电、里程钻取等专题页。
4. 系统能力：通过“更多 / 设置 / 关于”承接导出、报告、主题、语言、模拟模式、实例连接和版本信息。

该方案不把 Stitch 页面仅当作视觉稿，而是当作需求证据：

- 用页面标题定义产品模块边界。
- 用页面文本定义用户可见数据字段和交互动作。
- 用页面之间的入口关系定义信息架构和钻取顺序。
- 用仓库现有 `app_mimo` 目录结构反推实现落点，形成后续开发的统一目标。

## User Stories

1. 作为已部署 TeslaMate 的 Tesla 车主，我希望在手机首页看到车辆是否在线、是否驾驶中、是否已锁车，这样我能第一时间确认车辆当前状态。
2. 作为车主，我希望在仪表盘看到电池百分比和预估续航，这样我能快速判断今天是否需要充电。
3. 作为车主，我希望在仪表盘看到最近一趟行程摘要，这样我不用进入列表也能快速回忆刚刚的出行情况。
4. 作为车主，我希望在仪表盘看到里程表、充电限制和软件版本，这样我能用一个页面掌握车辆关键静态信息。
5. 作为车主，我希望在仪表盘看到胎压和车内外温度，这样我能判断车辆安全状态和当前舒适性。
6. 作为车主，我希望在仪表盘看到近 7 日耗电趋势，这样我能感知近期能耗波动。
7. 作为车主，我希望点击上次行程直接进入行程详情，这样我能自然地从总览钻取到单次记录。
8. 作为车主，我希望浏览行程历史列表，这样我能按月回看每一次出行。
9. 作为车主，我希望每条行程记录显示起止地点、距离、时长、日期和电量变化，这样我能快速判断这次出行值不值得深看。
10. 作为车主，我希望某些高效率行程被突出标记，这样我能识别“开得最好”的样本。
11. 作为车主，我希望进入单次行程详情后看到路线、距离、时长、最高速度、均速、能耗和效率，这样我能完整复盘一次驾驶。
12. 作为车主，我希望在行程详情中查看速度、功率、海拔、车内温度、车外温度等曲线切换，这样我能理解影响能耗的上下文。
13. 作为车主，我希望知道某些详情曲线是“真实时间序列”还是“基于摘要的模拟数据”，这样我能正确理解数据可信度。
14. 作为车主，我希望浏览充电历史列表，这样我能回看不同地点和不同类型的充电记录。
15. 作为车主，我希望在充电历史里同时看到总电量、总费用和“当前正在充电”的卡片，这样我能兼顾历史回顾和实时监控。
16. 作为车主，我希望区分 AC 和 DC 充电，并看到功率和费用，这样我能比较不同充电方式的成本与效率。
17. 作为车主，我希望点击“正在充电”进入更实时的充电详情，这样我能在充电中判断是否需要调整充电目标。
18. 作为车主，我希望在单次充电详情看到电量变化、功率 / 电压 / 温度 / 电量曲线和阶段划分，这样我能理解一次充电过程的质量。
19. 作为车主，我希望在充电详情中看到恒流段、恒压段、涓流段的划分，这样我能理解不同 SoC 区间的充电行为。
20. 作为车主，我希望查看热力图页面中的高频时间段、常去目的地和路线排行，这样我能从生活习惯角度理解车辆使用模式。
21. 作为车主，我希望查看续航分析中的“预估 vs 实际”偏差和影响因素，这样我能知道为什么现实续航和系统估算不同。
22. 作为车主，我希望看到综合续航评级和与其他车主的相对位置，这样我能判断自己的续航表现处于什么水平。
23. 作为车主，我希望查看能耗分析页面中的平均能耗、评级、趋势、分布和优化建议，这样我能针对性改善驾驶习惯。
24. 作为车主，我希望在能耗分析中看到同车型对比，这样我能判断自己的表现是车辆问题还是驾驶问题。
25. 作为车主，我希望查看电池健康页面中的健康度、容量衰减、循环统计和温度分布，这样我能长期跟踪电池状态。
26. 作为车主，我希望得到电池维护建议，例如减少高温快充、控制常用充电上限，这样我能更长久地维护电池寿命。
27. 作为车主，我希望查看待机耗电详情页面中的总损耗、平均功率、耗电来源和优化建议，这样我能减少车辆静置期间的隐性耗电。
28. 作为车主，我希望把哨兵模式、第三方插件唤醒、温度预调节等待机耗电原因区分开来，这样我能采取具体措施。
29. 作为车主，我希望查看里程钻取页面中的年度总里程、月度趋势、场景分布和最长里程日，这样我能用更宏观的视角理解用车模式。
30. 作为车主，我希望从里程钻取点击月份再点击具体日期，这样我能从年度趋势逐步钻取到具体行程。
31. 作为车主，我希望在“更多”页面中集中看到统计、热力图、效率、续航、能耗、电池健康、时间线和报告入口，这样我不用在主导航中塞入过多 tab。
32. 作为车主，我希望在“更多”页面看到年度报告 PDF、CSV/JSON 导出和固件版本入口，这样我能同时完成查阅和分享。
33. 作为车主，我希望在设置页输入服务器地址和 API Token，并可测试连接和保存，这样我能完成首次连接和后续换服。
34. 作为车主，我希望在设置页切换语言、主题和模拟模式，这样我能在真实环境和开发调试场景之间自由切换。
35. 作为车主，我希望在设置页看到当前实例的连接状态，这样我能知道应用当前是否真的连接到 TeslaMate。
36. 作为车主，我希望在关于页看到车辆基础信息、应用版本、平台技术栈和开源组件，这样我能快速确认产品构成和版本上下文。
37. 作为开发者，我希望页面信息架构优先围绕四个主入口组织，这样 Android、iOS、Web 可以尽量共用一套产品认知。
38. 作为开发者，我希望页面中所有分析能力都能映射到明确的数据契约和计算逻辑，这样实现不会因为“设计稿只有样子没有口径”而走偏。
39. 作为开发者，我希望当某些曲线或地图暂时缺少完整原始数据时，系统明确使用降级策略和提示文案，这样功能可以先上线但不误导用户。
40. 作为产品负责人，我希望这份 PRD 能直接映射到现有 `app_mimo` 的模块边界，这样后续排期、测试和验收都可以围绕同一份文档推进。

## Implementation Decisions

- 采用四主入口信息架构作为白色简约版的规范形态：`仪表盘 / 行程 / 充电 / 更多`。
- 将 `Tesla Data Tracker App` 与 `MateLink 仪表盘 (中文导航栏)` 视为同一类 Dashboard 变体，不单独拆成新模块。
- 将 Stitch 页面集中的“控制”痕迹视为旧变体或探索稿，不作为当前白色简约版的主导航规范。
- 以 `app_mimo` 为主落地方向，统一产品需求映射到三端目录结构：
  Android 对应 `data / domain / ui / notification / widget`；
  iOS 对应 `App / Core / Features / Widget`；
  Web 对应 `api / pages / components / store`。
- 仪表盘模块负责聚合车辆实时快照，而不承担复杂分析计算；复杂分析一律下沉到专题页。
- 行程与充电都采用“两级主链路”：列表页承接筛选和摘要，详情页承接单次记录曲线与导出。
- “当前充电”被视为充电历史链路中的实时态，而不是独立一级导航模块。
- 热力图、续航分析、能耗分析、电池健康、待机耗电、里程钻取统一归入“分析专题模块”，避免更多页面沦为杂项列表。
- 更多菜单页不仅是入口页，还承担“当前车辆摘要 + 分组导航 + 部分关键数值预览”的信息中转角色。
- 设置页的最小能力集固定为：服务器地址、API Token、连接测试、保存、语言切换、主题切换、模拟模式。
- 设置页以单实例体验为默认形态，但实现层要保留未来扩展到多实例/多车库的空间。
- 关于页保持“产品与技术透明”定位，展示版本、技术栈、依赖和车辆摘要，不做营销型内容。
- 数据契约上需要明确区分三类数据：
  实时状态快照、
  列表级摘要数据、
  详情级时间序列或可回放曲线。
- 对于 Stitch 页面中明确标示“模拟数据 — 基于摘要”的详情页，产品上允许先交付降级图表，但必须标出数据性质，不能让用户误以为这是完整采样曲线。
- 所有分析页应优先显示结论和对比，再显示原始曲线或图表，符合移动端快速消费场景。
- 数值表现遵循“数据优先”原则：电量、续航、能耗、费用、功率、效率等关键信息优先出现在首屏和卡片顶部。
- 多语言与主题切换属于产品内建能力，而不是后续美化项；白色简约风是当前 Stitch 项目的主视觉基准。
- 车辆状态、行程、充电、分析结果和设置项都要支持 Mock 模式，以便离线演示和开发联调。
- 报告与导出能力先定义为 PDF 年度报告和 CSV/JSON 数据导出，不把更复杂的云分享或协作纳入当前范围。
- 页面内地图能力在产品上是必须的，但实现上允许按平台降级：
  Android 可优先高德；
  iOS 可先用 MapKit 或后续切高德；
  Web 可先保证基础地图承载与轨迹可视化。
- 通知、Widget、时间线等系统能力不由 Stitch 页面完整定义，但必须与本 PRD 保持数据口径一致，不能各端各说各话。
- 产品数据基线应与现有 MateLink/TeslaMate 语义一致，包括车辆快照、驱动记录、充电记录、电池健康、软件版本、费用、效率与统计聚合。

## Testing Decisions

- 好的测试应该验证用户可见行为和稳定的数据口径，而不是验证某个界面内部如何布局、某个状态变量叫什么名字，或某个图表库内部如何绘制。
- 最优先测试的模块应是数据契约与分析逻辑，因为这些模块决定三端页面是否展示同一事实。
- 应重点测试以下模块：
  仪表盘快照聚合模块；
  行程列表与行程详情映射模块；
  充电列表、当前充电与充电详情映射模块；
  续航偏差、能耗评级、电池健康、待机耗电、里程钻取等分析计算模块；
  设置页连接测试与 Mock 模式切换模块；
  导出/报告入口的可见性与参数拼装模块。
- 对导航应做流程级验证：
  从仪表盘进入行程详情；
  从充电历史进入当前充电或单次充电详情；
  从更多页面进入各分析专题；
  从里程钻取继续钻取月度和日期。
- 对数据降级策略应做显式测试：
  有完整时间序列时展示真实曲线；
  只有摘要时展示降级图并标识“模拟数据”；
  无地图或无导出数据时展示明确占位和提示，而不是空白页。
- 对多语言和主题切换应做用户行为测试，保证界面切换不会破坏主要数据卡片、单位显示和底部导航文案。
- 对连接设置应测试成功连接、失败连接、空 Token、非法地址、局域网地址和 Mock 模式切换等典型场景。
- 就当前仓库现状看，Android 有一定单元测试基础，但 iOS 和 Web 的测试覆盖较薄，因此建议优先把“共享数据口径测试”建立起来，再逐步补 UI 流程验证。
- 在现有代码风格下，测试优先围绕以下思路补齐：
  API 响应到领域模型的映射；
  领域模型到 UI 展示项的转换；
  分析页结论数字的计算一致性；
  导航入口是否连接到正确页面。

## Out of Scope

- 直接车辆控制能力，例如解锁、空调控制、远程命令执行。
- Apple Watch、CarPlay、Android Auto 的完整产品定义。
- App Store 上架包装、营销页、截图运营物料。
- 社交分享、云端协作、团队驾驶分析等多人能力。
- 超出 Stitch 当前页面集合之外的全新一级模块。
- 将所有平台一次性做成完全同构实现；本 PRD 统一产品口径，但允许平台有阶段性差异。
- 复杂商业化、订阅体系、账号体系改造。
